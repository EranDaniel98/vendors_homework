# מאגר פגיעויות — בניית שכבת גישה חכמה

**שפה:** TypeScript
**זמן משוער:** 3–5 שעות

---

## הרקע

אחד מלקוחות Deloitte מנהל מאגר מידע פנימי של פגיעויות אבטחה — מערכת ותיקה שנבנתה לפני שנים על ידי צוות שכבר לא קיים. המאגר מתעד פגיעויות ידועות (CVEs) ומשייך כל אחת מהן לספק התוכנה הרלוונטי, ומשמש מדי יום צוותי אבטחה, מנהלי סיכונים ואנליסטים.

הבעיה? הנתונים יושבים בקבצים טקסטואליים שמורים על שרת פנימי. אין API, אין ממשק חיפוש, ואין דרך נוחה לשאול שאלות על הנתונים מבלי לפתוח את הקבצים ידנית.

הצוות רוצה לחבר את המאגר הזה לסוכן AI פנימי שיאפשר לאנליסטים לשאול שאלות בשפה טבעית — "כמה פגיעויות קריטיות עדיין פתוחות?", "אילו CVEs נמצאו ב-Linux Kernel בשנה האחרונה?", "מה ה-CVSS score של Log4Shell?" — ולקבל תשובות מיידיות.

הפתרון שנבחר: לבנות **MCP Server** שעוטף את הקבצים הקיימים ומחשוף אותם כ-tools שכל LLM תואם-MCP יכול להשתמש בהם.

## פורמט הנתונים הקיים

שני קבצים טקסטואליים ממתינים לך בתיקיית הפרויקט. הפורמט פותח פנימית ולא מבוסס על JSON, CSV, או כל סטנדרט מוכר — זה פורמט pipe-delimited מותאם אישית שנולד מאילוצים של מערכת ישנה.

כל קובץ מתחיל בבלוק metadata שמתאר את מבנה העמודות, ואחריו שורות הנתונים עצמן.

### vendors.db
קובץ זה מכיל את ספקי התוכנה הרשומים במערכת. ראה קובץ [`vendors.db`](./vendors.db) המצורף.

### vulnerabilities.db
קובץ זה מכיל את הפגיעויות הרשומות. כל פגיעות משויכת לספק אחד באמצעות שדה `vendor_id`. ראה קובץ [`vulnerabilities.db`](./vulnerabilities.db) המצורף.

**כמה דברים שכדאי לדעת על הפורמט:**
- שדה `affected_versions` מתאר טווח גרסאות כטקסט חופשי
- שדה `vendor_id` בקובץ הפגיעויות מפנה לשדה `id` בקובץ הספקים
- שדה `status` יכול להיות `open` או `patched`
- שדה `VERSION` ב-metadata קיים מסיבה — ייתכן שהפורמט ישתנה בעתיד
- המאגר צפוי לגדול — הפתרון צריך לעבוד גם עם אלפי רשומות

## מה צריך לבנות

### שלב א׳ — MCP Server

בנה MCP Server בטייפסקריפט תוך שימוש ב-SDK הרשמי (`@modelcontextprotocol/sdk`).

השרת צריך לטעון את שני הקבצים בהפעלה, לפרסר את ה-metadata בצורה דינמית, ולהחזיק את הנתונים בזיכרון לשאילתות מהירות.

**לגבי הכלים שתחשוף — אתה מקבל חופש מלא לעצב אותם.** חשוב על מה שאנליסט אבטחה אמיתי היה רוצה לשאול, ועצב את ה-tools בהתאם. הכלים צריכים לאפשר ל-LLM לנווט במאגר בצורה טבעית — לחפש, לסנן לפי חומרה או סטטוס, ולהצליב מידע בין ספקים לפגיעויות.

השרת צריך להיות ניתן לחיבור דרך Claude Desktop או כל MCP client תואם-פרוטוקול.

**פלט נדרש:**
- קוד מלא וניתן להרצה
- קובץ `README.md` עם הוראות הגדרה ותיאור כל כלי שמימשת
- התייחסות קצרה בקובץ ה-README להחלטות עיצוב שקיבלת ולמה

---

### שלב ב׳ — Agent Client *(רשות)*

אם יש לך גישה ל-LLM API, בנה שכבת agent מעל ה-MCP Server שמאפשרת לשאול שאלות בשפה טבעית ולקבל תשובות.

הסוכן צריך לקבל שאלה מהמשתמש — דרך CLI או ממשק פשוט — לקרוא לכלים הנכונים, ולסנתז תשובה סופית. שאלות שדורשות כמה קריאות לכלים צריכות לעבוד גם הן.

**אין מחויבות לספק API Key משלך.** אפשרויות חינמיות שעובדות היטב עם MCP:

- **Claude Desktop** — מתחבר ישירות ל-MCP Servers, ללא API key בכלל
- **Ollama** — הרצת מודלים מקומית (Llama, Mistral, Qwen ועוד)
- **Groq** — tier חינמי עם תמיכה מלאה ב-tool use
- **Google Gemini API** — tier חינמי נדיב עם function calling
